# Database context pack

> The hash-ledger migration walkthrough lives in `README.md`. This pack owns the
> agent-facing rules: what to edit, what never to touch, and how to review database PRs.

## The one rule: edit `supabase/src/**` only

| File / directory                | Status        | Rule                                                 |
| ------------------------------- | ------------- | ---------------------------------------------------- |
| `supabase/src/schemas/`         | Source        | Edit here                                            |
| `supabase/src/views/`           | Source        | Edit here                                            |
| `supabase/src/functions/`       | Source        | Edit here                                            |
| `supabase/src/policies/`        | Source        | Edit here                                            |
| `supabase/src/grants/`          | Source        | Edit here                                            |
| `supabase/migrations/`          | **Generated** | Never hand-edit                                      |
| `supabase/.migration-hash.json` | Ledger        | Never hand-edit — regenerated by `pnpm db:migration` |
| `src/lib/types/supabase.ts`     | **Generated** | Never hand-edit — regenerate with `pnpm db:types`    |

**Never rename or move files under `supabase/src/`** — the generator treats a renamed
file as a deleted object plus a new one and re-emits brand-new DDL, which will break
idempotency.

## Migration flow (summary — canonical steps in `README.md`)

1. Edit the relevant file(s) under `supabase/src/`.
2. Run `pnpm db:migration --name=describe_the_change` to generate the migration and
   update the ledger.
3. Run `pnpm db:migration:check` to verify the ledger is consistent.
4. Commit the source change, the generated migration, and the updated ledger together
   in one atomic commit.

For Codex on Windows: use `corepack pnpm db:migration --name=<name>`.

## Every new public table needs three things

1. `enable row level security` — without this the REST API can see all rows for any role.
2. At least one RLS policy — document the intended access pattern in the policy name.
3. `grant` statements — without these the Data API returns empty results even with RLS.

Add a pgTAP test in `supabase/tests/` for every new policy. See
[testing.md](testing.md) for the pgTAP layer details.

## One primary object per source file

Keep one table, view, or function as the primary subject of each source file.
Auxiliary objects (triggers, supporting functions) that belong to it can live in
the same file, but avoid grouping unrelated objects.

## Generator emit order (name files so dependencies sort first)

`pnpm db:migration` emits objects by folder in a fixed `SOURCE_ORDER` (schemas →
indexes → views → functions → policies → triggers → grants → comments) and, **within
each folder, alphabetically by filename** (`generate-migration.ts:33-42,62-64`). An
object another depends on must emit first, so:

- **Same folder:** a plain view that selects from a base matview in `views/` must sort
  **after** it — name the files accordingly. #406 used `league_ats_base` (base matview)
  with `league_ats_fav_dog` / `league_ats_home_away` / `league_ats_team` (dependents)
  precisely so the base sorts first; the originally-proposed `league_game_ats` /
  `league_team_ats` names would have emitted a dependent before the base and failed on
  first apply.
- **Cross folder:** views emit **before** functions, so a view must not call a function
  your same change introduces — from empty the function doesn't exist yet, which the
  ADR-0012 drift guard (`verify-src-reproduces-migrations.ts`) catches even though the
  real migration chain works (the function pre-exists from an earlier migration). Inline
  the logic, or land the function in a prior migration.

## Reviewing a database PR

Database PRs typically produce diffs of 500–5000+ lines because two large generated
files appear in every diff:

- **`src/lib/types/supabase.ts`** (often 4000+ lines): fully generated by
  `pnpm db:types`. Schema-only changes often reformat multiline union types to
  single-line equivalents. **Skip this file in review** — it proves nothing about
  correctness. If it changed, confirm the commit message says `pnpm db:types` was run.
- **`supabase/migrations/*.sql`** (generated DDL): the emitted migration is a
  faithful representation of the diff between source states. **Skip line-by-line
  review of the migration file** — review the source that generated it instead.

### What to actually review

```sh
# Scope the diff to the real review surface
git diff master...branch -- supabase/src supabase/tests supabase/.migration-hash.json
```

Check:

- `supabase/src/**` — is the SQL correct and minimal? Does a new table have RLS,
  grants, and a clear policy?
- `supabase/tests/**` — is there a pgTAP test for every new policy or function?
- `supabase/.migration-hash.json` — does it look freshly regenerated (not hand-edited)?

If the diff only touches `supabase.ts` and `supabase/migrations/` with no source
changes, that is a types-regeneration or reformatting commit — not a logic change.

## Validation for changes here

```sh
pnpm db:migration:check          # verify ledger consistency
npx supabase test db             # pgTAP (requires Docker + local Supabase)
pnpm test:integration            # integration tests against local Supabase
pnpm db:types                    # regenerate supabase.ts after schema changes
```

> **In a worktree, `pnpm db:reset:local` (prod-clone) can't run** — its clone step
> (`cloneDb.ts`) needs `SUPABASE_DB_URL_PROD`, which is absent from worktree `.env*`.
> Verify schema with `db:push:local` + a pgTAP fixture; run prod-data count-checks from
> the main checkout. See the
> [db-migration skill](../../.claude/skills/db-migration/SKILL.md).
